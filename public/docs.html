Auth Service API

Multi-tenant authentication microservice with email/password, OAuth2, magic links, passkeys, TOTP 2FA, and JWT session management.

Multi-tenant

Complete user/session isolation per client. One instance serves many apps.

Email / Password

Signup and login with bcrypt (cost 12), email verification, password reset.

OAuth2

Google, GitHub, Microsoft, Apple with PKCE-secured flows.

Magic Links

Passwordless email-based login. Creates accounts on first use.

Passkeys

WebAuthn/FIDO2 browser-native passwordless authentication.

TOTP 2FA

Time-based one-time passwords (RFC 6238) with QR setup.

JWT Tokens

HS256 or RS256 (JWKS). 15-min access + 7-day refresh rotation.

REST + gRPC

Dual protocol support. Go validator package for downstream services.

Quick Start Integration

Get your app authenticated in 4 steps:

Register your app as a client

Call the admin API to create a new tenant. Save the returned api_key and jwt_secret.

curl -X POST https://authservice.ayushojha.com/api/admin/clients \
  -H "Content-Type: application/json" \
  -H "X-Admin-Key: YOUR_ADMIN_KEY" \
  -d '{
    "name": "My App",
    "slug": "my-app",
    "allowed_origins": ["https://myapp.com"]
  }'

Add auth to your frontend

Send requests to the auth endpoints with the X-API-Key header.

// Signup
const res = await fetch("https://authservice.ayushojha.com/api/auth/signup", {
  method: "POST",
  headers: {
    "Content-Type": "application/json",
    "X-API-Key": "YOUR_API_KEY",
  },
  credentials: "include",  // for cookie-based refresh tokens
  body: JSON.stringify({
    email: "user@example.com",
    password: "securepassword123",
    display_name: "Jane Doe"
  })
});
const { access_token, user } = await res.json();

Validate JWTs in your backend

Use the jwt_secret to validate tokens locally. No network call to the auth service needed.

import "github.com/Ayush10/authentication-service/pkg/jwtvalidator"

validator := jwtvalidator.New(jwtvalidator.Config{
    Secret: os.Getenv("AUTH_JWT_SECRET"),
})

// Use as middleware
http.Handle("/api/", validator.Middleware(apiRouter))

// Or validate manually
claims, err := validator.Validate(tokenString)
if err != nil {
    // invalid or expired token
}
userID := claims.UserID()  // sub claim
email := claims.Email
role := claims.Role
const jwt = require("jsonwebtoken");

function validateToken(token) {
  try {
    return jwt.verify(token, process.env.AUTH_JWT_SECRET, {
      algorithms: ["HS256"],
    });
  } catch (err) {
    return null;
  }
}

// Express middleware
function authMiddleware(req, res, next) {
  const auth = req.headers.authorization;
  if (!auth?.startsWith("Bearer ")) return res.status(401).json({ error: "missing token" });
  const claims = validateToken(auth.slice(7));
  if (!claims) return res.status(401).json({ error: "invalid token" });
  req.user = claims;
  next();
}
import jwt, os

JWT_SECRET = os.environ["AUTH_JWT_SECRET"]

def validate_token(token: str) -> dict | None:
    try:
        return jwt.decode(token, JWT_SECRET, algorithms=["HS256"])
    except jwt.InvalidTokenError:
        return None

# FastAPI dependency
async def get_current_user(authorization: str = Header(...)):
    if not authorization.startswith("Bearer "):
        raise HTTPException(401)
    claims = validate_token(authorization[7:])
    if not claims:
        raise HTTPException(401)
    return claims

Set the JWT secret in your backend environment

AUTH_JWT_SECRET=your_jwt_secret_from_step_1

That's it. Your backend validates tokens locally with zero latency. No calls back to the auth service.

Register a Client (Tenant)

Every app that uses the auth service must register as a client. This creates isolated user pools, sessions, and JWT signing keys.

POST /api/admin/clients X-Admin-Key

Request Body

{
  "name": "My Application",
  "slug": "my-app",                         // unique, URL-safe identifier
  "allowed_origins": ["https://myapp.com"],  // CORS origins
  "webhook_url": "https://myapp.com/hooks"    // optional event webhook
}

Response 201 Created

{
  "client": {
    "id": "fe24972b-68fd-4670-904c-55496033d079",
    "name": "My Application",
    "slug": "my-app",
    "allowed_origins": ["https://myapp.com"],
    "status": "active",
    "created_at": "2025-01-15T10:30:00Z",
    "updated_at": "2025-01-15T10:30:00Z"
  },
  "api_key": "77f61b0b47df7c...",    // shown ONCE - save this!
  "jwt_secret": "f41deb9c1476..."  // shown ONCE - save this!
}
⚠
Save both values immediately. The api_key and jwt_secret are only returned once. The API key is stored as a SHA-256 hash and cannot be retrieved later. If lost, you must rotate.

A verification email is sent automatically if the email provider (Resend) is configured. The refresh token is set as an HttpOnly cookie named auth_refresh.

Login

POST /api/auth/login X-API-Key

Request

{
  "email": "user@example.com",
  "password": "securepassword123"
}

Response 200 OK

Same format as signup response. If the user has TOTP enabled, you'll get a 2FA challenge instead:

{
  "requires_2fa": true,
  "two_factor_token": "temp-2fa-token-uuid",
  "two_factor_methods": ["totp"]
}

When you receive this, prompt the user for their TOTP code and call TOTP Verify to complete login.

Refresh Token

POST /api/auth/refresh X-API-Key

Rotates the refresh token and returns a new access token. Send the request with credentials: "include" so the auth_refresh cookie is sent automatically.

Response 200 OK

{
  "access_token": "eyJhbGciOiJIUzI1NiIs...",
  "token_type": "Bearer",
  "expires_in": 900
}
ℹ
Token rotation: Each refresh token is single-use. Using it issues a new refresh token and revokes the old one. If a revoked token is reused, the entire session is invalidated as a security measure.

Logout

POST /api/auth/logout X-API-Key

Revokes the session and clears the auth_refresh cookie. Send with credentials: "include".

Response 200 OK

{ "message": "logged out" }

Magic Link Authentication

Passwordless login via email. If the user doesn't exist, an account is created automatically.

Send Magic Link

POST /api/auth/magic-link/send X-API-Key

Request

{ "email": "user@example.com" }

Rate limited to 5 per hour per IP. Token expires in 15 minutes.

Verify Magic Link

GET /api/auth/magic-link/verify?token=TOKEN Public

The user clicks this link from their email. The service authenticates them and either:

  • Returns JSON with access_token (API client), or
  • Redirects to your app's login page with ?access_token=TOKEN (browser flow)

OAuth2 Social Login

Supported providers: google, github, microsoft, apple

ℹ
OAuth providers must be configured with their respective client IDs and secrets in the auth service's environment. Only configured providers will be available.

Begin OAuth Flow

GET /api/auth/oauth/{provider} X-API-Key

Redirects the user to the provider's consent screen. The flow uses PKCE with a signed state payload for security.

Frontend Example

// Redirect user to OAuth provider
window.location.href =
  "https://authservice.ayushojha.com/api/auth/oauth/google"
  + "?api_key=YOUR_API_KEY";
⚠
For OAuth, you can pass the API key as a query parameter ?api_key=... since redirect flows can't set custom headers.

OAuth Callback

GET /api/auth/oauth/{provider}/callback Public

Handled automatically. After successful authentication, the user is redirected to your app:

https://yourapp.com/login?access_token=eyJhbGci...

The auth_refresh cookie is also set. If the OAuth email matches an existing user, accounts are linked automatically.

POST /api/auth/totp/setup X-API-Key Bearer Token

Generates a TOTP secret. Show the QR code to the user in their authenticator app.

Response

  "qr": "data:image/png;base64,iVBOR..."

POST /api/auth/totp/enable X-API-Key Bearer Token

Confirm setup with a valid code from the authenticator app.

POST /api/auth/totp/verify X-API-Key

Called after login returns requires_2fa: true.

{

  "code": "123456"

Returns the standard login response with access_token and user profile.

Disable TOTP

POST /api/auth/totp/disable X-API-Key Bearer Token

{ "code": "123456" }

Passkeys (WebAuthn / FIDO2)

Browser-native passwordless authentication using platform authenticators (Touch ID, Windows Hello) or security keys.

Register a Passkey

POST /api/auth/passkey/register/begin X-API-Key Bearer Token

Returns WebAuthn creation options. Pass these to navigator.credentials.create().

Request

{ "name": "My MacBook" }  // optional friendly name
POST /api/auth/passkey/register/finish X-API-Key Bearer Token

Send the attestation response from the browser back to the server to complete registration.

Login with Passkey

POST /api/auth/passkey/login/begin X-API-Key

Returns assertion options. Pass these to navigator.credentials.get().

Request (optional)

{ "email": "user@example.com" }  // optional, narrows allowed credentials
POST /api/auth/passkey/login/finish X-API-Key

Send the assertion response from the browser. Returns standard login response with access_token.

Frontend Passkey Flow Example

const API = "https://authservice.ayushojha.com";
const headers = {
  "X-API-Key": API_KEY,
  "Content-Type": "application/json",
};

// 1. Begin login
const opts = await fetch(`${API}/api/auth/passkey/login/begin`, {
  method: "POST", headers, credentials: "include",
  body: JSON.stringify({})
}).then(r => r.json());

// 2. Browser authenticator prompt
const credential = await navigator.credentials.get({
  publicKey: opts.publicKey
});

// 3. Finish login
const result = await fetch(`${API}/api/auth/passkey/login/finish`, {
  method: "POST", headers, credentials: "include",
  body: JSON.stringify(credential)
}).then(r => r.json());

console.log(result.access_token);

JWT Validation (Backend Integration)

The auth service issues JWTs that your backend validates locally. No network call to the auth service is needed.

Access Token Claims

{
  "sub": "user-uuid",          // user ID
  "email": "user@example.com",
  "email_verified": true,
  "role": "user",
  "client_id": "client-uuid",  // tenant ID
  "jti": "unique-token-id",
  "iat": 1700000000,
  "exp": 1700000900            // 15 minutes
}

Token Modes

ModeAlgorithmValidationBest For
v1_hs256HMAC-SHA256Shared jwt_secretPrivate backends, microservices
v2_jwksRSA RS256JWKS endpoint (public key)Public APIs, third-party integrations

Go: Using the jwtvalidator Package

go get github.com/Ayush10/authentication-service/pkg/jwtvalidator

HS256 Mode (default)

import "github.com/Ayush10/authentication-service/pkg/jwtvalidator"

validator := jwtvalidator.New(jwtvalidator.Config{
    Secret: os.Getenv("AUTH_JWT_SECRET"),  // jwt_secret from client registration
})

// As HTTP middleware (injects claims into context)
protectedMux := http.NewServeMux()
protectedMux.HandleFunc("/api/profile", profileHandler)
http.Handle("/api/", validator.Middleware(protectedMux))

// Manual validation
claims, err := validator.Validate(tokenString)
if err != nil {
    // token invalid, expired, or tampered
    http.Error(w, "unauthorized", 401)
    return
}

// Access claims
userID := claims.UserID()       // sub
email := claims.Email           // email
role := claims.Role             // role
verified := claims.EmailVerified
clientID := claims.ClientID

RS256 Mode (JWKS)

validator := jwtvalidator.New(jwtvalidator.Config{
    JWKSURL:         "https://authservice.ayushojha.com/.well-known/jwks.json?client_id=YOUR_ID",
    ClientID:        "your-client-uuid",
    RefreshInterval: 5 * time.Minute,  // how often to refresh JWKS
})

Other Languages

Use any JWT library that supports HS256. The validation logic is the same:

  1. Extract the Bearer token from the Authorization header
  2. Verify the signature using your jwt_secret with HS256
  3. Check that exp has not passed
  4. Read sub (user ID), email, role, client_id from claims

JWKS Endpoint

GET /.well-known/jwks.json

Returns the JSON Web Key Set for RS256 token validation. Requires either:

  • X-API-Key header, or
  • ?client_id=UUID query parameter

Response

{
  "keys": [
    {
      "kty": "RSA",
      "kid": "key-id",
      "use": "sig",
      "alg": "RS256",
      "n": "modulus...",
      "e": "AQAB"
    }
  ]
}

Token Validation via gRPC

If you can't use the JWT secret directly (e.g., non-Go services that need the auth service to validate), you can call the TokenService:

// Proto definition
message ValidateTokenRequest {
  string access_token = 1;
  string api_key = 2;
}

message ValidateTokenResponse {
  bool valid = 1;
  string user_id = 2;
  string email = 3;
  string role = 4;
  bool email_verified = 5;
  string client_id = 6;
  string error = 7;
}

Session Modes

The auth service supports two ways to deliver refresh tokens:

Cookie Mode (default)

The refresh token is set as an HttpOnly, Secure cookie named auth_refresh. Best for browser-based apps.

  • Automatically sent with credentials: "include"
  • Protected from XSS (JavaScript can't access it)
  • Requires proper CORS/origin configuration

Token Mode

Pass ?session_mode=token on login/signup to receive the refresh token in the JSON response body instead of a cookie.

Best for mobile apps, CLI tools, or environments without cookie support.

POST /api/auth/login?session_mode=token

The response will include:

{ ..., "refresh_token": "abc123..." }

Send it back in subsequent refresh calls as:

{ "refresh_token": "abc123..." }
